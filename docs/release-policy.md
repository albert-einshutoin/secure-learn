# Release Policy

Secure Learn uses Semantic Versioning. `VERSION`, the Git tag, and the changelog version must match. A release tag is created only from `main` after required checks have passed.

## Release gate

A release is publishable only when all of the following are true:

1. Source, unit, integration, Compose, generated-document, and Suricata rule checks pass.
2. A fresh stack with empty volumes completes SIEM bootstrap and the telemetry-backed S3 E2E check.
3. The application, Suricata, IPS helper, and non-root host publisher make up the four images in the release evidence. The IPS helper must pass a networkless runtime check covering both the installed `iptables` binary and an isolated NFQUEUE rule lifecycle with only `NET_ADMIN` added. The host publisher must pass a networkless, capability-free `socat` startup check.
4. Trivy reports no fixed HIGH or CRITICAL vulnerabilities across all four images. Unfixed findings are intentionally excluded from this blocking report and must be assessed separately when base-image updates are reviewed.
5. Syft produces four SPDX JSON SBOM files, one for each image.
6. SHA-256 checksums and four GitHub artifact attestations are generated for the release evidence.

The release workflow creates a draft, attaches every scan report and SBOM, and only then publishes it so GitHub can lock the release and its tag. Source archives are generated by GitHub from the immutable tag. Container images are rebuilt locally from digest-pinned bases; this project does not currently publish a registry image.

On Docker Desktop, an `internal: true` bridge is intentionally unreachable from host port forwarding. Secure Learn keeps the attack and data planes internal and uses the minimal host publisher as an application-layer TCP boundary; the publisher is non-root, read-only, capability-free, has IP forwarding disabled, and is the only service class attached to the non-internal `host_access` network.

`scripts/release_artifacts.sh` mounts the Docker socket into the digest-pinned Trivy and Syft containers so they can inspect locally built images. Docker socket access is effectively host-equivalent Docker daemon access, so this exception is for trusted release tooling only. It is never part of `scripts/learn`, learner lab workflows, or untrusted contribution execution; run it only in the protected release job or on a maintainer-controlled workstation after reviewing the pinned scanner images.

## Support and disclosure

Only the newest minor release receives routine fixes. Security reports follow [SECURITY.md](../SECURITY.md). A compromised or materially incorrect release is not overwritten: publish a new patch version and document the impact and migration in `CHANGELOG.md`.
