S13 / L7 / 初級

L7 DNS観測

Docker embedded DNSのservice discoveryを観測し、内部偵察と運用依存を理解する。

2/5 基礎良好 WhitehatSRE

抽象的に何を学ぶか

L7 DNS観測は、単発の攻撃手順ではなく、DNSが単なる名前解決ではなく、偵察、障害、service discoveryの証跡になることを理解する。

具体例

  • Observe: getent hosts、dig、/etc/resolv.confでDocker DNSを見る。
  • Map: service名、IP、networkを図にする。
  • Explain: Docker embedded DNSがどこで動き、Suricataで安定観測しづらい理由を説明する。

読み方

初学者の見方

  • 最初に「抽象的に何を学ぶか」と「目的」を読み、何を守る演習かを言葉にする。
  • 事前準備と安全境界を確認してから、実行コマンドを上から順に一つずつ実行する。
  • 合格証跡の先頭項目「service名とIPの対応」を残し、ログまたはHTTP結果と対応付ける。

経験者の深掘り

  • 攻撃/障害の再現だけで止めず、検知漏れ、誤検知、本番移行時の追加統制を洗い出す。
  • ツール活用と観測ポイントを読み、どの信号が意思決定に使えるかを評価する。
  • 世界レベル課題の先頭項目「CoreDNS log、DNS tunneling、DGA検知を追加する。」をPRレビュー観点へ落とす。

レビューで見ること

  • 第三者が再現できるように、対象、時刻、payload、検索条件、判断理由を揃える。
  • Whitehat/SRE/Backendの観点で、誰がownerになり、どこでrollbackするかを決める。
  • セルフレビューの回答をincident reportまたはremediation PRの本文へ転記する。

学習フロー図

1 理解

抽象説明、目的、具体例で守る資産と失敗条件を把握する。

2 準備

Docker、対象範囲、安全境界を確認して外部環境へ出さない。

3 実行

Hands-on Flowと実行コマンドを小さく進める。

4 観測

HTTP、ログ、検知イベント、メトリクスを対応付ける。

5 証跡化

合格証跡としてservice名とIPの対応を残す。

6 深掘り

世界レベル課題を本番運用、PR、runbookへ変換する。

目的

DNSが単なる名前解決ではなく、偵察、障害、service discoveryの証跡になることを理解する。

世界レベル評価

基礎から実務中級の土台を作る構成です。高度領域はS16以降の専門シナリオへ接続します。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS13のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Observe

    getent hosts、dig、/etc/resolv.confでDocker DNSを見る。

  2. Map

    service名、IP、networkを図にする。

  3. Explain

    Docker embedded DNSがどこで動き、Suricataで安定観測しづらい理由を説明する。

  4. Detect

    異常なservice discoveryをSIEMで見つける設計を考える。

  5. Operate

    DNS障害時の切り分け手順を作る。

実行コマンド

docker exec -it soc-lab-kali /bin/bash
/scripts/s13_l7_dns_observe.sh
getent hosts app db elasticsearch
cat /etc/resolv.conf
dig app

合格証跡

  • service名とIPの対応
  • resolver設定
  • DNSログがない場合の証跡不足説明
  • 実運用で取るべきDNS telemetry

ツール活用

Tool使い方
getent/dig名前解決の実結果を確認する。
Docker DNS127.0.0.11とservice discoveryを理解する。
Kibana実運用ではDNS logをSIEM化する前提で検索設計する。
Network mapservice nameとIPの対応を可視化する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「service名とIPの対応」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • L7 DNS観測で守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

Whitehat

攻撃を許可範囲に限定し、再現性、検知証跡、影響範囲を明確にする。

SRE

SLO、可用性、レイテンシ、復旧判断、incident severityへ接続する。

Backend

サービス露出、ログ、HTTP contract、依存関係への影響を確認する。

世界レベルへ足す課題

  • CoreDNS log、DNS tunneling、DGA検知を追加する。
  • Kubernetes Service DNSとNetworkPolicyを接続する。
  • DNS failureをincident drill化する。
