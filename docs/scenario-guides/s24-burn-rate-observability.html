Burn-rate Alert・RED/USE・Alert Fatigue

SLI/SLO、error budget、multi-window burn-rate、RED/USE metrics、alert fatigueを実測結果へ接続する。

4/5 実務上級 SREBackend

抽象的に何を学ぶか

Observabilityはdashboardを増やすことではなく、顧客影響を判断する質問に答えられる信号を作ることです。

具体例

  • 5分burn-rateが高いが1時間では低い場合、pageではなくticketにする。
  • RED metricsでAPI影響を、USE metricsでnode/resource飽和を分ける。
  • user_idをmetric labelに入れるとhigh cardinalityで監視基盤を壊す。

目的

SLO違反を検知し、alertの緊急度、owner、actionを判断できるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

Hands-on Flow

  1. Baseline

    load testでp95とerror rateを取得する。

  2. Budget

    SLOとerror budget消費を計算する。

  3. Alert

    multi-window burn-rateのpage条件とticket条件を決める。

  4. Correlate

    log、metric、traceをどの質問に使うか整理する。

  5. Reduce noise

    alert fatigueを減らすdedup、silence、routingを設計する。
