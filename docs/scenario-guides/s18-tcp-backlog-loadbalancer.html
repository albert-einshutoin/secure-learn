TCP再送・SYN backlog・Load Balancer

TCP再送、SYN backlog、conntrack、L4/L7 load balancer logを、SLO劣化の根拠として扱う。

4/5 実務上級 SREWhitehat

抽象的に何を学ぶか

TCPやload balancerの障害は、アプリログに到達しないことがあります。抽象的には、requestがアプリに届く前にどこで落ちたかを観測点ごとに切り分けます。

具体例

  • SYN backlogが溢れるとアプリにはrequestが来ず、clientはtimeoutや再送を見る。
  • L7 LBはHTTP statusを出せるが、L4 LBはconnection resetやtimeoutの証跡が中心になる。
  • conntrack枯渇はpodやappではなくnode/network層のcapacity問題になる。

目的

L4/L7の観測点を整理し、アプリ障害、LB障害、network capacityを分けてincident判断できるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

Hands-on Flow

  1. Baseline

    app直通とedge proxy経由のhealth latencyを測る。

  2. Observe

    ss、tcpdump、nginx access log、app access logを比較する。

  3. Model

    SYN backlog、conntrack、timeoutのfailure modeを図にする。

  4. Decide

    scale out、rate limit、LB timeout変更、rollbackの条件を決める。

  5. Report

    どの観測点に証跡があり、どこにないかを明記する。

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: サービス露出、ログ、HTTP契約との関係を確認する。

世界レベルへ足す課題

  • real LB/NLB/ALB/Envoy logsを取り込む。
  • packet lossとretransmissionをblackbox SLIへ接続する。
  • load balancer config変更をcanary化する。
