Detection Engineering・EDR Case Management

Sigma、YARA、Suricata、MITRE ATT&CK、false positive tuning、event normalization、SIEM query、case managementを扱う。

4/5 実務上級 WhitehatSRE

抽象的に何を学ぶか

Detection engineeringはalertを増やすことではなく、攻撃仮説、telemetry、正規化、誤検知調整、case判断を一つの運用にすることです。

具体例

  • Suricata ruleをMITRE techniqueへ対応させ、何を検知できて何を検知できないかを書く。
  • Sigma/YARAはvendor非依存の検知意図として保存し、SIEM queryへ変換する。
  • false positiveが多いruleはthreshold、allowlist、context enrichmentで調整する。

目的

検知ルールを作るだけでなく、case triage、owner、severity、closure reasonまで運用できるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS30のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Hypothesis

    攻撃仮説と必要telemetryを定義する。

  2. Normalize

    source.ip、user.name、process.name、event.actionなどのfieldを揃える。

  3. Map

    MITRE ATT&CK techniqueとruleを対応させる。

  4. Tune

    false positive/false negativeを評価する。

  5. Case

    case management形式でtimeline、severity、closureを書く。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「attack hypothesisとtelemetry要件」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • Detection Engineering・EDR Case Managementで守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: サービス露出、ログ、HTTP契約との関係を確認する。

世界レベルへ足す課題

  • SigmaをElastic/KQLへ変換して実検索する。
  • case management systemへticketを作る。
  • detection-as-codeのunit testを追加する。
