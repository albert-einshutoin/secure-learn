Terraform・Drift Detection・OPA

Terraform module、state、drift、OPA/Conftest、CI plan reviewを、infra変更の安全装置として扱う。

4/5 実務上級 SREWhitehat

抽象的に何を学ぶか

IaCは環境構築の自動化だけではなく、変更意図、差分、policy違反、driftをレビュー可能にする仕組みです。

具体例

  • 0.0.0.0/0 ingressやpublic bucketをpolicyで拒否する。
  • prod/stageのstateを分け、module inputだけで差分を表現する。
  • console手変更をdriftとして検出し、緊急対応後にcodeへ戻す。

読み方

初学者の見方

  • 最初に「抽象的に何を学ぶか」と「目的」を読み、何を守る演習かを言葉にする。
  • 事前準備と安全境界を確認してから、実行コマンドを上から順に一つずつ実行する。
  • 合格証跡の先頭項目「plan review comment」を残し、ログまたはHTTP結果と対応付ける。

経験者の深掘り

  • 攻撃/障害の再現だけで止めず、検知漏れ、誤検知、本番移行時の追加統制を洗い出す。
  • ツール活用と観測ポイントを読み、どの信号が意思決定に使えるかを評価する。
  • 世界レベル課題の先頭項目「terraform plan JSONとOPA/Regoを実行する。」をPRレビュー観点へ落とす。

レビューで見ること

  • 第三者が再現できるように、対象、時刻、payload、検索条件、判断理由を揃える。
  • Whitehat/SRE/Backendの観点で、誰がownerになり、どこでrollbackするかを決める。
  • セルフレビューの回答をincident reportまたはremediation PRの本文へ転記する。

学習フロー図

1 理解

抽象説明、目的、具体例で守る資産と失敗条件を把握する。

2 準備

Docker、対象範囲、安全境界を確認して外部環境へ出さない。

3 実行

Hands-on Flowと実行コマンドを小さく進める。

4 観測

HTTP、ログ、検知イベント、メトリクスを対応付ける。

5 証跡化

合格証跡としてplan review commentを残す。

6 深掘り

世界レベル課題を本番運用、PR、runbookへ変換する。

目的

IaC変更をplan、policy、state、drift、rollbackの5点でレビューできるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS23のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。
  • 実cloud、BGP、CDN、組織policyは変更せず、設計レビュー、サンプルログ、ローカル検証に限定する。

Hands-on Flow

  1. Read

    Terraform風サンプルを読み、危険なdiffを指摘する。

  2. Policy

    OPA/Conftest相当の禁止条件を文章化する。

  3. CI

    plan outputにsecurity reviewerが何をコメントするかを書く。

  4. Drift

    手変更を検出した時の復旧方針を決める。

  5. Promote

    dev/stage/prodの環境分離と承認を設計する。

合格証跡

  • plan review comment
  • policy-as-code禁止条件
  • state/drift/environment分離の説明
  • 例外承認と期限

ツール活用

Tool使い方
Terraform planresource差分、destroy、public exposureを見る。
OPA/Conftest組織policyをCIで機械的に評価する。
State backendlocking、encryption、environment separationを確認する。
Drift detection手変更とcode差分を検出する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「plan review comment」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • Terraform・Drift Detection・OPAで守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

Whitehat

攻撃を許可範囲に限定し、再現性、検知証跡、影響範囲を明確にする。

SRE

SLO、可用性、レイテンシ、復旧判断、incident severityへ接続する。

Backend

サービス露出、ログ、HTTP contract、依存関係への影響を確認する。

世界レベルへ足す課題

  • terraform plan JSONとOPA/Regoを実行する。
  • drift detectionをscheduled CIへ入れる。
  • cloud asset inventoryとIaC stateを照合する。
