S11 / L5 / 中級

L5セッション圧迫

不完全HTTP sessionを保持し、timeout、connection limit、SLO影響を観測する。

3/5 実務中級 SREBackend

抽象的に何を学ぶか

L5セッション圧迫は、単発の攻撃手順ではなく、大量攻撃ではなく少数接続のリソース消費を理解し、proxy/app/IDSの責務分担を説明する。

具体例

  • Baseline: /healthとload gateで正常時latencyを測る。
  • Execute: /scripts/s11_l5_session_stress.shで不完全sessionを作る。
  • Observe: 接続保持時間、Suricata alert、/health latencyを見る。

目的

大量攻撃ではなく少数接続のリソース消費を理解し、proxy/app/IDSの責務分担を説明する。

世界レベル評価

基礎から実務中級の土台を作る構成です。高度領域はS16以降の専門シナリオへ接続します。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS11のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Baseline

    /healthとload gateで正常時latencyを測る。

  2. Execute

    /scripts/s11_l5_session_stress.shで不完全sessionを作る。

  3. Observe

    接続保持時間、Suricata alert、/health latencyを見る。

  4. Triage

    timeout、keepalive、rate limit、connection limitを分けて説明する。

  5. Improve

    reverse proxyやapp serverのtimeout設計を提案する。

実行コマンド

scripts/sre_smoke.sh
docker exec -it soc-lab-kali /bin/bash
/scripts/s11_l5_session_stress.sh
scripts/load_hands_on_tests.sh
docker exec soc-lab-suricata tail -30 /var/log/suricata/fast.log

合格証跡

  • 接続保持数と保持時間
  • 攻撃中のlatency/failure
  • timeout設定案
  • 正常ユーザー影響の判断

ツール活用

Tool使い方
nc不完全なHTTP sessionを保持する。
Suricatasession pressure/DOS系alertを確認する。
SRE smoke/loadlatencyとavailabilityを定量化する。
Proxy configheader timeout、keepalive、max connectionsを設計する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「接続保持数と保持時間」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • L5セッション圧迫で守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

  • Whitehat: このシナリオでは補助観点として扱う。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: 入力検証、契約、テスト、エラー安全性を確認する。

世界レベルへ足す課題

  • slowloris系の安全な再現とproxy防御を追加する。
  • event loop lagやconnection pool exhaustionをmetrics化する。
  • 負荷と攻撃を区別するalert設計を追加する。
