S6 / OS / 中級

権限昇格

sudo/su/SUID探索をAuditdで記録し、最小権限と異常昇格の判断を学ぶ。

実行型ラボ 3/5 標準 WhitehatSRE

実行形式

実行型ラボ

同梱Docker環境、攻撃スクリプト、検証スクリプトを使って再現と観測を実行します。

教材範囲の自己評価

基礎概念と観測の土台を扱う教材です。点数は教材範囲の内部目安であり、技能や本番適合性の認定ではありません。

抽象的に何を学ぶか

権限昇格は、単発の攻撃手順ではなく、権限昇格を実行できたで終わらせず、許可、証跡、影響、封じ込め、権限設計まで説明する。

具体例

  • Prepare: auditdとprivilege escalation rulesを有効化する。
  • Execute: sudo、su、SUID列挙を安全な範囲で実行する。
  • Observe: audit.logとauth.logでeuid=0、auid、commandを確認する。

読み方

初学者の見方

  • 最初に「抽象的に何を学ぶか」と「目的」を読み、何を守る演習かを言葉にする。
  • 事前準備と安全境界を確認してから、実行コマンドを上から順に一つずつ実行する。
  • 合格証跡の先頭項目「昇格前後のuid/euid」を残し、ログまたはHTTP結果と対応付ける。

経験者の深掘り

  • 攻撃/障害の再現だけで止めず、検知漏れ、誤検知、本番移行時の追加統制を洗い出す。
  • ツール活用と観測ポイントを読み、どの信号が意思決定に使えるかを評価する。
  • 発展課題の先頭項目「GTFOBins調査、container breakout防止、Linux capabilityを追加する。」をPRレビュー観点へ落とす。

レビューで見ること

  • 第三者が再現できるように、対象、時刻、payload、検索条件、判断理由を揃える。
  • Whitehat/SRE/Backendの観点で、誰がownerになり、どこでrollbackするかを決める。
  • セルフレビューの回答をincident reportまたはremediation PRの本文へ転記する。

目的

権限昇格を実行できたで終わらせず、許可、証跡、影響、封じ込め、権限設計まで説明する。

本番環境との差分

ローカル教材で得た証跡だけでは本番適合性を保証できません。組織固有の脅威モデル、権限、可用性、法令、変更管理で追加検証してください。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS6のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。
  • 攻撃payloadは教材内の明示されたendpoint、file、containerだけに向ける。

Hands-on Flow

  1. Prepare

    auditdとprivilege escalation rulesを有効化する。

  2. Execute

    sudo、su、SUID列挙を安全な範囲で実行する。

  3. Observe

    audit.logとauth.logでeuid=0、auid、commandを確認する。

  4. Decide

    正当運用、要調査、不正昇格に分類する。

  5. Improve

    sudoers最小化、MFA、session recordingを提案する。

実行コマンド

sudo auditctl -l | grep privilege
sudo whoami
find / -perm -4000 2>/dev/null | head
sudo ausearch -k privilege_escalation -i | head -50
sudo aureport --auth --summary

合格証跡

  • 昇格前後のuid/euid
  • auditdのexecve証跡
  • 正当性判断とowner確認
  • sudoers改善案とrollback手順

ツール活用

Tool使い方
sudo/su昇格イベントを意図的に発生させる。
findSUID/SGIDの攻撃面を列挙する。
Auditdeuid、auid、execveを保存する。
sudoers reviewNOPASSWDや広すぎる権限を評価する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「昇格前後のuid/euid」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • 権限昇格で守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

Whitehat

攻撃を許可範囲に限定し、再現性、検知証跡、影響範囲を明確にする。

SRE

SLO、可用性、レイテンシ、復旧判断、incident severityへ接続する。

Backend

サービス露出、ログ、HTTP contract、依存関係への影響を確認する。

本番導入へ追加する課題

  • GTFOBins調査、container breakout防止、Linux capabilityを追加する。
  • PAM/MFA/session recordingの運用設計を追加する。
  • Kubernetes RBAC/PodSecurityと接続する。
