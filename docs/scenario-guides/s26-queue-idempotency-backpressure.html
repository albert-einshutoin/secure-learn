S26 / Distributed Systems / 上級

Queue・Idempotency・Backpressure

Kafka/PubSub/Temporal相当の非同期処理を、retry、backoff、idempotency、backpressureとして設計する。

4/5 実務上級 BackendSRE

抽象的に何を学ぶか

分散システムでは「1回送れば1回処理される」と仮定できません。抽象的には、重複、遅延、順序入れ替わり、部分失敗を前提に契約を作ります。

具体例

  • producer retryで同じmessageが2回届いても、idempotency keyで二重登録を防ぐ。
  • consumerが遅い時にqueue backlogを見てscale、shed、degradeを判断する。
  • Temporal workflowではactivity retryとcompensationを明示する。

目的

queue/cache依存のfailure modeをAPI契約、SLO、運用runbookへ落とせるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

Hands-on Flow

  1. Start

    Redis profileを起動し、queueの代替としてlist/backlogを扱う。

  2. Retry

    同じ処理を複数回送る時のidempotency条件を書く。

  3. Backpressure

    backlog増加時の429、degrade、scale判断を作る。

  4. Consistency

    leader election、replication、shardingの選択理由を書く。

  5. Report

    partial failure前提のrunbookを作る。

  • Backend: 入力検証、契約、テスト、エラー安全性を確認する。
