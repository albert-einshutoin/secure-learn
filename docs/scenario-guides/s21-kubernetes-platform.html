Kubernetes本番運用・Platform Guardrails

Helm、Kustomize、Operator/CRD、Admission、RBAC、NetworkPolicy、PodSecurity、HPA/VPA、upgradeを本番運用として扱う。

4/5 実務上級 SREWhitehatBackend

抽象的に何を学ぶか

Kubernetes本番運用はyaml適用ではなく、危険な変更を入れない仕組み、壊れても戻せる仕組み、upgradeしても契約を守る仕組みを作ることです。

具体例

  • Admissionでprivileged podやlatest imageを拒否する。
  • RBACでsecretを読めるServiceAccountを限定し、NetworkPolicyでeast-west通信を絞る。
  • CRD/Operator upgrade前にAPI deprecationとrollback条件を確認する。

読み方

初学者の見方

  • 最初に「抽象的に何を学ぶか」と「目的」を読み、何を守る演習かを言葉にする。
  • 事前準備と安全境界を確認してから、実行コマンドを上から順に一つずつ実行する。
  • 合格証跡の先頭項目「manifestごとの安全条件説明」を残し、ログまたはHTTP結果と対応付ける。

経験者の深掘り

  • 攻撃/障害の再現だけで止めず、検知漏れ、誤検知、本番移行時の追加統制を洗い出す。
  • ツール活用と観測ポイントを読み、どの信号が意思決定に使えるかを評価する。
  • 世界レベル課題の先頭項目「kind/minikubeでAdmission policyを実行する。」をPRレビュー観点へ落とす。

レビューで見ること

  • 第三者が再現できるように、対象、時刻、payload、検索条件、判断理由を揃える。
  • Whitehat/SRE/Backendの観点で、誰がownerになり、どこでrollbackするかを決める。
  • セルフレビューの回答をincident reportまたはremediation PRの本文へ転記する。

学習フロー図

1 理解

抽象説明、目的、具体例で守る資産と失敗条件を把握する。

2 準備

Docker、対象範囲、安全境界を確認して外部環境へ出さない。

3 実行

Hands-on Flowと実行コマンドを小さく進める。

4 観測

HTTP、ログ、検知イベント、メトリクスを対応付ける。

5 証跡化

合格証跡としてmanifestごとの安全条件説明を残す。

6 深掘り

世界レベル課題を本番運用、PR、runbookへ変換する。

目的

Kubernetesをdeployment targetではなく、policyとoperabilityを持つplatformとして説明できるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS21のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Read

    k8s/baseのDeployment、Service、HPA、NetworkPolicy、StatefulSetを読む。

  2. Validate

    scripts/k8s_static_check.shで最低限のguardrailを検査する。

  3. Design

    Helm/Kustomize/Admission/RBAC/PodSecurityの追加設計を書く。

  4. Operate

    cluster upgrade、multi-cluster、service meshのrisk tableを作る。

  5. Review

    manifest diff reviewで拒否すべき変更を列挙する。

実行コマンド

scripts/learning_phase.sh start p12
scripts/k8s_static_check.sh
find k8s/base -type f -maxdepth 2 -print
sed -n "1,220p" docs/runbooks/kubernetes-operations.md
scripts/world_class_hands_on_check.sh kubernetes

合格証跡

  • manifestごとの安全条件説明
  • Admission/RBAC/NetworkPolicyの設計案
  • upgradeとrollback手順
  • service mesh/multi-cluster導入時のリスク評価

ツール活用

Tool使い方
Kustomize/Helm環境差分とrelease unitを管理する。
Admission Controllerpolicy違反をclusterへ入る前に拒否する。
RBAC/NetworkPolicy/PodSecurityidentity、network、runtimeの境界を作る。
HPA/VPAcapacityとresource requestを運用する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「manifestごとの安全条件説明」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • Kubernetes本番運用・Platform Guardrailsで守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

Whitehat

攻撃を許可範囲に限定し、再現性、検知証跡、影響範囲を明確にする。

SRE

SLO、可用性、レイテンシ、復旧判断、incident severityへ接続する。

Backend

入力検証、認証認可、DB/API契約、エラー安全性、テストで再発を止める。

世界レベルへ足す課題

  • kind/minikubeでAdmission policyを実行する。
  • OPA Gatekeeper/KyvernoをCIとclusterに入れる。
  • service mesh mTLSとtraffic shiftingを実測する。
