SQLインジェクション

sqlmap/curlでSQLiを試行し、アプリが400で拒否し、DB queryがparameterizedであることを証明する。

実行型ラボ 4/5 発展 WhitehatBackend

実行形式

実行型ラボ

同梱Docker環境、攻撃スクリプト、検証スクリプトを使って再現と観測を実行します。

教材範囲の自己評価

複数の観点と証跡作成まで扱う発展教材です。点数は教材範囲の内部目安であり、技能や本番適合性の認定ではありません。

抽象的に何を学ぶか

SQLインジェクションは、単発の攻撃手順ではなく、攻撃の再現、検知、修正済み契約、DB integration testを一つのremediation evidenceにまとめる。

具体例

  • Prepare: docker compose up -d --buildでPostgreSQL込みの環境を起動する。
  • Attack: /scripts/s3_sqli.shでpayloadとsqlmapを実行する。
  • Defense: HTTP 400、sqli_attempt log、Suricata SQLI alertを確認する。

読み方

初学者の見方

  • 最初に「抽象的に何を学ぶか」と「目的」を読み、何を守る演習かを言葉にする。
  • 事前準備と安全境界を確認してから、実行コマンドを上から順に一つずつ実行する。
  • 合格証跡の先頭項目「SQLi payloadとHTTP 400結果」を残し、ログまたはHTTP結果と対応付ける。

経験者の深掘り

  • 攻撃/障害の再現だけで止めず、検知漏れ、誤検知、本番移行時の追加統制を洗い出す。
  • ツール活用と観測ポイントを読み、どの信号が意思決定に使えるかを評価する。
  • 発展課題の先頭項目「boolean/time/error/union/in-band/out-of-bandの観点でpayload coverageを増やす。」をPRレビュー観点へ落とす。

レビューで見ること

  • 第三者が再現できるように、対象、時刻、payload、検索条件、判断理由を揃える。
  • Whitehat/SRE/Backendの観点で、誰がownerになり、どこでrollbackするかを決める。
  • セルフレビューの回答をincident reportまたはremediation PRの本文へ転記する。

目的

攻撃の再現、検知、修正済み契約、DB integration testを一つのremediation evidenceにまとめる。

本番環境との差分

ローカル教材で得た証跡だけでは本番適合性を保証できません。組織固有の脅威モデル、権限、可用性、法令、変更管理で追加検証してください。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS3のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。
  • 攻撃payloadは教材内の明示されたendpoint、file、containerだけに向ける。

Hands-on Flow

  1. Prepare

    docker compose up -d --buildでPostgreSQL込みの環境を起動する。

  2. Attack

    /scripts/s3_sqli.shでpayloadとsqlmapを実行する。

  3. Defense

    HTTP 400、sqli_attempt log、Suricata SQLI alertを確認する。

  4. Backend proof

    UsersServiceのparameterized query testとDB integration testを実行する。

  5. Report

    CWE-89、影響、修正、残リスク、再発防止をPR形式で記録する。

実行コマンド

docker compose up -d --build
docker exec -it soc-lab-kali /bin/bash
/scripts/s3_sqli.sh
scripts/backend_hands_on_tests.sh

合格証跡

  • SQLi payloadとHTTP 400結果
  • UsersService unit testとDB integration test結果
  • Suricata/App logの検知証跡
  • エラーにSQL/stack traceが出ていない証跡

ツール活用

Tool使い方
sqlmap脆弱性探索ツールとして使うが、修正済みでは成立しないことを確認する。
curl単一payloadのHTTP statusとresponse shapeを確認する。
PostgreSQL integration test実DBに対してpayloadがデータを返さないことを検証する。
Suricata/Kibana攻撃試行そのものの検知証跡を保存する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「SQLi payloadとHTTP 400結果」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • SQLインジェクションで守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

Whitehat

攻撃を許可範囲に限定し、再現性、検知証跡、影響範囲を明確にする。

SRE

影響がある場合だけSLO、ログ量、復旧手順との差分を見る。

Backend

入力検証、認証認可、DB/API契約、エラー安全性、テストで再発を止める。

本番導入へ追加する課題

  • boolean/time/error/union/in-band/out-of-bandの観点でpayload coverageを増やす。
  • ORM/Repository境界、migration、transaction rollbackまで拡張する。
  • DASTをCIの安全なプロファイルで走らせ、誤検知triageを課題化する。
