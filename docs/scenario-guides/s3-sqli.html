S3 / L7 / 中級

SQLインジェクション

sqlmap/curlでSQLiを試行し、アプリが400で拒否し、DB queryがparameterizedであることを証明する。

4/5 実務上級 WhitehatBackend

目的

攻撃の再現、検知、修正済み契約、DB integration testを一つのremediation evidenceにまとめる。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

Hands-on Flow

  1. Prepare

    docker compose up -d --buildでPostgreSQL込みの環境を起動する。

  2. Attack

    /scripts/s3_sqli.shでpayloadとsqlmapを実行する。

  3. Defense

    HTTP 400、sqli_attempt log、Suricata SQLI alertを確認する。

  4. Backend proof

    UsersServiceのparameterized query testとDB integration testを実行する。

  5. Report

    CWE-89、影響、修正、残リスク、再発防止をPR形式で記録する。

実行コマンド

docker compose up -d --build
docker exec -it soc-lab-kali /bin/bash
/scripts/s3_sqli.sh
scripts/backend_hands_on_tests.sh
npm --prefix app test
npm --prefix app run test:integration

合格証跡

  • SQLi payloadとHTTP 400結果
  • UsersService unit testとDB integration test結果
  • Suricata/App logの検知証跡
  • エラーにSQL/stack traceが出ていない証跡

ツール活用

Tool使い方
sqlmap脆弱性探索ツールとして使うが、修正済みでは成立しないことを確認する。
curl単一payloadのHTTP statusとresponse shapeを確認する。
PostgreSQL integration test実DBに対してpayloadがデータを返さないことを検証する。
Suricata/Kibana攻撃試行そのものの検知証跡を保存する。

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: 影響がある場合だけSLO差分を確認する。
  • Backend: 入力検証、契約、テスト、エラー安全性を確認する。

世界レベルへ足す課題

  • boolean/time/error/union/in-band/out-of-bandの観点でpayload coverageを増やす。
  • ORM/Repository境界、migration、transaction rollbackまで拡張する。
  • DASTをCIの安全なプロファイルで走らせ、誤検知triageを課題化する。
