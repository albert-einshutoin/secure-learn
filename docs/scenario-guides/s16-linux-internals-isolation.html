Linux Internals・隔離境界

systemd、cgroups、namespaces、seccomp、capabilitiesを、containerの障害調査と防御境界として扱う。

4/5 実務上級 SREWhitehat

抽象的に何を学ぶか

Linux internalsは、プロセスがCPU、メモリ、ファイル、ネットワーク、syscall、権限をどう使っているかを読む訓練です。抽象的には、障害や侵害を「OS資源と隔離境界の変化」として説明します。

具体例

  • cgroupsでCPU/memory制限があると、アプリは5xxではなくlatency悪化やOOMとして壊れる。
  • namespace内のrootはhost rootではないが、capabilityが広いと危険なsyscallやnetwork操作が可能になる。
  • strace/lsof/ssで、HTTP requestがsocket、file descriptor、syscall待ちとして見える。

目的

Linuxの資源管理と隔離機構を、container security、障害調査、SRE runbookに接続できるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

Hands-on Flow

  1. Observe

    ps、ss、lsof、straceでappと通信の状態を見る。

  2. Inspect

    docker inspectでcap_drop、security_opt、read_only、resource設定を確認する。

  3. Explain

    namespace、cgroup、capability、seccompが守る範囲と守れない範囲を表にする。

  4. Tune safely

    kernel tuning値は変更せず、見るべき値と変更時のrollback条件を整理する。

  5. Report

    OS起因障害とアプリ起因障害を分けた調査メモを作る。

実行コマンド

scripts/learning_phase.sh start p10
docker exec learn-toolbox ss -tan
docker exec learn-toolbox sh -lc 'curl -fsS http://app:3000/health && lsof -iTCP -P -n | head'
docker inspect soc-lab-app | jq '.[0].HostConfig.CapDrop, .[0].HostConfig.SecurityOpt'
scripts/world_class_hands_on_check.sh linux

合格証跡

  • process、socket、fd、syscallを対応付けた記録
  • capabilities/seccomp/cgroupの防御境界説明
  • kernel tuning候補とrollback条件
  • OS起因とアプリ起因の切り分け表

ツール活用

Tool使い方

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: サービス露出、ログ、HTTP契約との関係を確認する。

世界レベルへ足す課題

  • eBPF/perf flamegraphを本番相当環境で扱う。
  • seccomp profileとcapability最小化をCIで検査する。
  • kernel CVE時のnode drain、upgrade、rollback runbookを作る。
