S10 / L4 / 中級

L4 TCP状態・フラグ異常

SYN/connect/FIN/NULL/XMAS scanの違いを観測し、L4状態と検知ルールを理解する。

3/5 実務中級 WhitehatSRE

抽象的に何を学ぶか

L4 TCP状態・フラグ異常は、単発の攻撃手順ではなく、TCP flagとconnection stateを説明し、露出ポート最小化と検知閾値の設計に繋げる。

具体例

  • Prepare: Kaliからtargetへ到達できる状態を確認する。
  • Execute: /scripts/s10_l4_tcp_state.shで複数scanを実行する。
  • Observe: nmap結果、Suricata fast.log、app access.logを比較する。

目的

TCP flagとconnection stateを説明し、露出ポート最小化と検知閾値の設計に繋げる。

世界レベル評価

基礎から実務中級の土台を作る構成です。高度領域はS16以降の専門シナリオへ接続します。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS10のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Prepare

    Kaliからtargetへ到達できる状態を確認する。

  2. Execute

    /scripts/s10_l4_tcp_state.shで複数scanを実行する。

  3. Observe

    nmap結果、Suricata fast.log、app access.logを比較する。

  4. Triage

    L4で止まった通信とL7まで届いた通信を区別する。

  5. Improve

    露出ポート、firewall、IDS thresholdを見直す。

実行コマンド

docker exec -it soc-lab-kali /bin/bash
/scripts/s10_l4_tcp_state.sh
nmap -sS -p 3000 app
nmap -sF -sN -sX -p 3000 app
docker exec soc-lab-suricata tail -30 /var/log/suricata/fast.log

合格証跡

  • scan方式ごとのnmap結果
  • Suricata alertとtimestamp
  • L7到達有無
  • 露出ポート削減案

ツール活用

Tool使い方
nmapscan方式ごとの応答差を見る。
SuricataTCP flag anomalyやscan alertを見る。
App access.logconnect scanがL7まで到達したか確認する。
Firewall/NetworkPolicyL4で制限すべき通信を設計する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「scan方式ごとのnmap結果」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • L4 TCP状態・フラグ異常で守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: サービス露出、ログ、HTTP契約との関係を確認する。

世界レベルへ足す課題

  • normal baselineとの差分検知を追加する。
  • Kubernetes NetworkPolicyやcloud security groupの実証へ広げる。
  • service mesh/proxyのL4 timeout設計を比較する。
