S29 / Secure SDLC / 上級

Supply Chain・SBOM・Secure Release

SBOM、SAST、DAST、SCA、secret scanning、artifact signing、provenance、release noteをrelease gateへ統合する。

4/5 実務上級 WhitehatBackendSRE

抽象的に何を学ぶか

Supply chain securityは、コードだけでなく依存、CI、artifact、署名、配布、脆弱性開示までを攻撃面として扱います。

具体例

  • SCAは依存CVE、SASTはコードpattern、DASTは実行中API、SBOMは部品表を担当する。
  • artifact signingで、mainのどのcommitからimageが作られたか検証可能にする。
  • secret scanningで漏洩を早期検知し、rotationまでを手順化する。

目的

secure releaseをCI結果、SBOM、scan、署名、rollback、advisoryの証跡で説明できるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS29のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Inventory

    package-lockとDocker imageからcomponent inventoryを作る。

  2. Scan

    npm audit、secret scanning、SAST/DAST/SCAの役割を整理する。

  3. Provenance

    commit、CI run、artifact、release noteを紐づける。

  4. Sign

    artifact signingとverificationの導入手順を書く。

  5. Disclose

    security advisory、CVE/CVSS、responsible disclosureを確認する。

実行コマンド

npm --prefix app audit --omit=dev --audit-level=high
scripts/world_class_hands_on_check.sh supply-chain
sed -n "1,220p" SECURITY.md
sed -n "1,220p" docs/templates/vulnerability-remediation-pr.md

合格証跡

  • SBOM/SAST/DAST/SCA coverage map
  • CI run、commit、artifactの対応
  • secret rotation手順
  • security advisoryとrelease note案

ツール活用

Tool使い方
SBOMcomponent、version、license、vulnerabilityを一覧化する。
SAST/DAST/SCAコード、実行中API、依存を別々に検査する。
Secret scanning漏洩検知とrotationを行う。
Signing/provenanceartifactの出自を検証する。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「SBOM/SAST/DAST/SCA coverage map」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • Supply Chain・SBOM・Secure Releaseで守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: 入力検証、契約、テスト、エラー安全性を確認する。

世界レベルへ足す課題

  • CycloneDX/Syft、cosign、SLSA provenanceを実行する。
  • container image scanとlicense policyをCIに入れる。
  • security advisory dry-runを実施する。
