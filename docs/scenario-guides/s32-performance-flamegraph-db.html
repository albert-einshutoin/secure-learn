Performance Engineering・Flamegraph・DB Query Plan

profiling、load test、p95/p99、connection pool、N+1、GC、CPU flamegraph、memory leak、DB index/query planを扱う。

4/5 実務上級 BackendSRE

抽象的に何を学ぶか

Performance engineeringは速くする作業ではなく、どのresourceがどのSLOを壊しているかを証拠で特定し、変更の副作用を測る作業です。

具体例

  • p50は正常でもp99が悪い場合、一部requestのDB queryやlockが原因かもしれない。
  • N+1は平均負荷では見えにくく、データ量増加で急にSLOを壊す。
  • flamegraphはCPU hot path、heap profileはmemory leak、query planはDB bottleneckを見る。

目的

性能劣化をCPU、memory、GC、DB、network、poolに分解し、改善PRの証跡を作れるようにする。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

Hands-on Flow

  1. Measure

    load testでp95/p99とfailureを取る。

  2. Hypothesize

    CPU、memory、DB、connection pool、networkの仮説を立てる。

  3. Profile

    ラボではNode build/testとOS観測を使い、実務ではflamegraph/query planへ拡張する。

  4. Fix safely

    index、pagination、pool、cache、backpressureの副作用を比較する。

  5. Verify

    before/afterのSLOとrollback条件を記録する。

実行コマンド

REQUESTS=120 CONCURRENCY=8 scripts/load_hands_on_tests.sh
npm --prefix app test
scripts/world_class_hands_on_check.sh performance

合格証跡

  • p95/p99とthroughput
  • bottleneck仮説と反証
  • query plan/index/pool設計
  • before/afterとrollback条件

Whitehat / SRE / Backend 観点

  • Whitehat: このシナリオでは補助観点として扱う。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: 入力検証、契約、テスト、エラー安全性を確認する。
