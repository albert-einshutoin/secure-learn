S14 / 横断/SRE / 上級

SREインシデント対応

攻撃やDB停止をSLO/incident responseへ接続し、MTTD/MTTRとpostmortemを作る。

4/5 実務上級 SREBackendWhitehat

抽象的に何を学ぶか

SREインシデント対応は、単発の攻撃手順ではなく、セキュリティイベントをサービス影響へ翻訳し、severity、暫定対応、恒久対応を判断する。

具体例

  • Baseline: scripts/sre_smoke.shとload gateで正常時SLOを記録する。
  • Inject: DoSまたはRUN_CHAOS=1 scripts/incident_drill.shで障害を注入する。
  • Detect: /health/ready、Suricata、Fail2ban、Kibanaを同時に見る。

目的

セキュリティイベントをサービス影響へ翻訳し、severity、暫定対応、恒久対応を判断する。

世界レベル評価

実務上級に近い構成です。証跡、修正、運用判断まで扱えます。

事前準備

  • Docker DesktopまたはDocker Engineが起動している。
  • docker compose config -q が通り、対象service/profileを説明できる。
  • 対象はローカルラボとS14のHTMLに書かれた範囲だけに限定する。

安全境界

  • 第三者環境、公共IP、許可のないcloud accountには実行しない。
  • payload、scan、負荷はHTMLに書かれたローカル対象へ限定する。

Hands-on Flow

  1. Baseline

    scripts/sre_smoke.shとload gateで正常時SLOを記録する。

  2. Inject

    DoSまたはRUN_CHAOS=1 scripts/incident_drill.shで障害を注入する。

  3. Detect

    /health/ready、Suricata、Fail2ban、Kibanaを同時に見る。

  4. Respond

    severity、incident commander、customer impact、rollback判断を記録する。

  5. Learn

    postmortemで再発防止とownerを明記する。

実行コマンド

scripts/sre_smoke.sh
REQUESTS=100 CONCURRENCY=10 scripts/load_hands_on_tests.sh
RUN_CHAOS=1 scripts/incident_drill.sh
docker compose ps
docker exec soc-lab-fail2ban fail2ban-client status

合格証跡

  • SLO baselineと障害中の差分
  • MTTD/MTTR
  • severityと顧客影響
  • postmortem action item

ツール活用

Tool使い方
SRE smoke/load scriptsavailabilityとp95 latencyを測る。
readiness probe依存関係障害を/healthと分離する。

Postmortem templatetimeline、impact、action itemを残す。

観測ポイント

  • 実行コマンドの開始時刻、対象、終了時刻を記録する。
  • HTTP status、アプリログ、検知ログ、メトリクスのどれで成功/失敗を証明するかを決める。
  • 合格証跡として「SLO baselineと障害中の差分」を第三者が追える形で残す。

よくある失敗

  • 攻撃が成功したことと、検知や防御が成功したことを混同する。
  • source、timestamp、request、検索条件を残さず、後から再現できない。
  • ローカルで動いた事実だけで完了扱いにし、本番で必要な追加統制を説明しない。

セルフレビュー

  • SREインシデント対応で守りたい資産と失敗条件を一文で説明できるか。
  • どのログ、テスト、メトリクスが判断根拠かを第三者が追えるか。
  • 本番導入時の追加統制、owner、rollback条件を言えるか。

Whitehat / SRE / Backend 観点

  • Whitehat: 攻撃を許可範囲に限定し、再現性と検知証跡を残す。
  • SRE: SLO、可用性、レイテンシ、復旧判断へ接続する。
  • Backend: 入力検証、契約、テスト、エラー安全性を確認する。

世界レベルへ足す課題

  • burn-rate multi-window alertを実装する。
  • canary/rollback/release freeze判断をdrill化する。
  • traces/metrics/logsの相関とdashboardを追加する。
