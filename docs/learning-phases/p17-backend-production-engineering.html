Backend Production Engineering

P18に進む前に、backend変更を運用・セキュリティ・互換性の3軸でreviewできる

BackendSREWhitehat

抽象的に何を学ぶか

Backend productionでは、ローカルで動くAPIではなく、schema変更、互換性、同時実行、負荷、障害時にも顧客契約を壊さないAPIを作ります。

具体例

  • migration: expand/contractで旧versionと新versionを同時に動かせる
  • race condition: 同時更新で二重処理やlost updateが起きないようtransaction境界を決める
  • pagination/API versioning: 大量データと互換性をcontract testで守る

読み方

初学者の見方

  • 最初に到達目標とカバー領域を読み、知らない用語を一つずつ潰す。
  • Docker実行のconfig、start、statusを順に実行し、どのprofileが動くか確認する。
  • 合格証跡の先頭項目「migration/rollback/API compatibilityのテスト設計がある」を残してから次フェーズへ進む。

経験者の深掘り

  • 自分の既存業務と照らし、欠けている運用、セキュリティ、テストの観点を洗い出す。
  • 実行コマンドを単に通すだけでなく、失敗時のログ、metric、rollback条件を確認する。
  • 次フェーズ判定「P18に進む前に、backend変更を運用・セキュリティ・互換性の3軸でreviewできる」をレビュー可能な証跡として説明する。

レビューで見ること

  • 前フェーズの未完了を放置せず、証跡、判断理由、残リスクを揃える。
  • Whitehat/SRE/Backendのどの役割で何を学んだかを一文で説明する。
  • 本番へ持ち込む場合の追加権限、owner、承認、監視、rollbackを定義する。

学習フロー図

1 理解

到達目標、カバー領域、具体例で学習対象を把握する。

2 準備

前フェーズ証跡、安全境界、Docker profileを確認する。

3 起動

learning_phase.sh start p17で環境を起動する。

4 実行

実行コマンドとHands-on Flowを順番に進める。

5 観測

service health、ログ、メトリクス、テスト結果を対応付ける。

6 判定

合格証跡と次フェーズ判定を満たした根拠を残す。

到達目標

  • schema migration/rollbackとAPI compatibilityを設計できる
  • transaction、race condition、pagination、rate limitをテスト観点に落とせる
  • DB index/query plan、N+1、connection poolを性能問題として説明できる

カバー領域

  • schema migration
  • rollback
  • transaction
  • race condition
  • pagination
  • API versioning
  • OpenAPI compatibility
  • contract testing
  • rate limit design
  • DB index
  • query plan

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p17 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。

Hands-on Flow

  • app testとOpenAPI contract testを実行する
  • migration rollback、pagination、API versioningのred/green test案を書く
  • load test結果からDB/APIのbottleneck仮説を立てる

Docker実行

scripts/learning_phase.sh config p17
scripts/learning_phase.sh start p17
scripts/learning_phase.sh status p17

実行コマンド

npm --prefix app test
npm --prefix app run test:integration
scripts/backend_hands_on_tests.sh
REQUESTS=120 CONCURRENCY=8 scripts/load_hands_on_tests.sh
sed -n '1,220p' docs/scenario-guides/s27-backend-migration-contract.html

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P18に進む前に、backend変更を運用・セキュリティ・互換性の3軸でreviewできる」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Backend Production Engineeringの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • migration/rollback/API compatibilityのテスト設計がある
  • race conditionとtransaction境界の説明がある
  • p95/p99、query plan、index、connection poolの改善仮説がある
