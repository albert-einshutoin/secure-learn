Linux Internals・Container Isolation

P11に進む前に、OS資源不足とアプリ障害をログだけで混同しない

SREWhitehat

抽象的に何を学ぶか

Linux internalsでは、アプリの障害や侵害をOSの資源管理、隔離、システムコール、権限境界として読む訓練をします。抽象的には「プロセスが何を許可され、何に詰まり、どの証跡を残すか」を説明する力です。

具体例

  • cgroups: containerにCPU/memory制限をかけた時、p95 latencyとOOMのどちらが先に悪化するかを観測する
  • namespaces/capabilities: rootに見えるprocessでもhost rootではない理由を説明し、cap_dropで攻撃面を減らす
  • strace/lsof/ss: API遅延時にopen file、socket、syscall待ちを切り分ける

到達目標

  • process、thread、file descriptor、socket、signalを運用証跡として読める
  • namespace、cgroup、capability、seccompがcontainer securityにどう効くか説明できる
  • strace、lsof、ss、tcpdump、perf/eBPFの使い分けを判断できる

カバー領域

  • systemd
  • cgroups
  • namespaces
  • seccomp
  • capabilities
  • eBPF
  • perf
  • strace
  • lsof
  • kernel tuning

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p10 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。

Hands-on Flow

  • learning toolboxでps、ss、lsof、straceを使い、appへのHTTP requestがsocket/syscallとして見えることを確認する
  • docker inspectでcapabilities、security_opt、resource制限を確認する
  • kernel tuning項目を変更せず、どの値を本番で見るべきかrunbookに整理する

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P11に進む前に、OS資源不足とアプリ障害をログだけで混同しない」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Linux Internals・Container Isolationの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • process/socket/syscall/file descriptorを紐づけたメモがある
  • container isolationで守れることと守れないことを分けて説明している
  • capability/seccomp/cgroupの改善案に影響範囲とrollback条件がある
