分散システム・性能・容量計画

P9に進む前に、性能劣化を設計・測定・運用判断へ接続できる

SREBackend

抽象的に何を学ぶか

分散システム・性能・容量計画では、個別のコマンド暗記ではなく、queue/cacheの障害がAPIとSLOへ与える影響を説明できるための判断軸を学びます。

具体例

  • 具体例: distributed profileを起動し、Redis疎通とapp healthを確認する
  • 具体例: load_hands_on_testsでp95を測定する
  • 具体例: queue/cache依存を追加する場合のfailure modeを設計する

読み方

初学者の見方

  • 最初に到達目標とカバー領域を読み、知らない用語を一つずつ潰す。
  • Docker実行のconfig、start、statusを順に実行し、どのprofileが動くか確認する。
  • 合格証跡の先頭項目「latency、throughput、error rate、resource limitを記録している」を残してから次フェーズへ進む。

経験者の深掘り

  • 自分の既存業務と照らし、欠けている運用、セキュリティ、テストの観点を洗い出す。
  • 実行コマンドを単に通すだけでなく、失敗時のログ、metric、rollback条件を確認する。
  • 次フェーズ判定「P9に進む前に、性能劣化を設計・測定・運用判断へ接続できる」をレビュー可能な証跡として説明する。

レビューで見ること

  • 前フェーズの未完了を放置せず、証跡、判断理由、残リスクを揃える。
  • Whitehat/SRE/Backendのどの役割で何を学んだかを一文で説明する。
  • 本番へ持ち込む場合の追加権限、owner、承認、監視、rollbackを定義する。

Gate 次フェーズ判定

P9に進む前に、性能劣化を設計・測定・運用判断へ接続できる

到達目標

  • queue/cacheの障害がAPIとSLOへ与える影響を説明できる
  • retry、timeout、backpressure、idempotencyを設計できる
  • p95/p99、connection pool、resource limitを容量計画に変換できる

カバー領域

  • Redis
  • queue
  • backpressure
  • idempotency
  • capacity planning

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p8 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。
  • 負荷試験は小さい値から始め、SLO悪化を確認したら停止して復旧を記録する。

Hands-on Flow

  • distributed profileを起動し、Redis疎通とapp healthを確認する
  • load_hands_on_testsでp95を測定する
  • queue/cache依存を追加する場合のfailure modeを設計する

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P9に進む前に、性能劣化を設計・測定・運用判断へ接続できる」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • 分散システム・性能・容量計画の到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • latency、throughput、error rate、resource limitを記録している
  • retry stormやqueue backlogの対策を説明している
  • capacity planningとrollback条件がある
