P19 / Principal / capstone

Detection/EDR・Secure SDLC・OSS Governance

修了。実務では本物のcloud account、production-like telemetry、cross-team incidentで継続訓練する

WhitehatSREBackend

抽象的に何を学ぶか

最終フェーズでは、技術的に検知・修正できるだけでなく、OSSとして安全に公開し、CVE/開示/ライセンス/レビュー規律まで運用します。

具体例

  • Detection: Sigma/YARA/SuricataをMITRE techniqueへ対応させ、false positiveを調整する
  • EDR: process tree、auditd、Sysmon相当telemetryからcase timelineを作る
  • OSS governance: advisory、CVE/CVSS、branch protection、license compliance、release noteを揃える

読み方

初学者の見方

  • 最初に到達目標とカバー領域を読み、知らない用語を一つずつ潰す。
  • Docker実行のconfig、start、statusを順に実行し、どのprofileが動くか確認する。
  • 合格証跡の先頭項目「MITRE mapping、SIEM query、case timeline、false positive判断がある」を残してから次フェーズへ進む。

経験者の深掘り

  • 自分の既存業務と照らし、欠けている運用、セキュリティ、テストの観点を洗い出す。
  • 実行コマンドを単に通すだけでなく、失敗時のログ、metric、rollback条件を確認する。
  • 次フェーズ判定「修了。実務では本物のcloud account、production-like telemetry、cross-team incidentで継続訓練する」をレビュー可能な証跡として説明する。

レビューで見ること

  • 前フェーズの未完了を放置せず、証跡、判断理由、残リスクを揃える。
  • Whitehat/SRE/Backendのどの役割で何を学んだかを一文で説明する。
  • 本番へ持ち込む場合の追加権限、owner、承認、監視、rollbackを定義する。

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p19 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。
  • 攻撃再現は教材内の明示されたendpoint、file、containerに限定する。

Hands-on Flow

  • S7/S15をcapstoneとして実行し、case management形式でtimelineを作る
  • detection ruleの誤検知、正規化、SIEM queryをレビューする
  • OSS release checklistを作り、branch protectionとsecurity advisory手順を確認する

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Detection/EDR・Secure SDLC・OSS Governanceの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

  • auditd/Sysmon/process tree/sandboxingの限界を説明している
  • security advisory、CVE/CVSS、license、release note、branch protectionの運用証跡がある
