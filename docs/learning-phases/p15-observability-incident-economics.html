Advanced Observability・Incident Economics

P16に進む前に、観測データをincident判断へ変換できる

SREBackend

抽象的に何を学ぶか

Advanced observabilityでは、見えるデータを増やすのではなく、顧客影響と運用判断に必要な信号だけをSLO、trace、log、alertへ落とします。

具体例

  • burn-rate: 5分と1時間の両方でerror budget消費を見てpage条件を決める
  • trace/log correlation: request_idでslow requestをDB queryや外部依存へ繋ぐ
  • high cardinality: user_idをlabelに入れず、log fieldとして保持する

到達目標

  • RED/USE metrics、tracing、logsを役割別に使い分けられる
  • multi-window burn-rate alertとalert fatigue削減を設計できる
  • high-cardinality labelのコストと運用影響を説明できる

カバー領域

  • Prometheus
  • Grafana
  • OpenTelemetry
  • tracing
  • RED metrics
  • USE metrics
  • log correlation
  • high cardinality
  • alert fatigue
  • burn-rate alert
  • error budget

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p15 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。
  • 負荷試験は小さい値から始め、SLO悪化を確認したら停止して復旧を記録する。

Hands-on Flow

  • Prometheus/Grafana/OpenTelemetry profileを起動する
  • load test結果をSLO/error budget/burn-rateへ変換する
  • alertをpage、ticket、dashboard annotationに分類する

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P16に進む前に、観測データをincident判断へ変換できる」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Advanced Observability・Incident Economicsの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • SLI/SLO/error budget/burn-rate条件が数値で書かれている
  • metric/log/traceをどの質問に使うか説明している
  • alert fatigueを減らすrouting、silence、dedup方針がある
