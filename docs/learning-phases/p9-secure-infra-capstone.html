抽象的に何を学ぶか

大手セキュアインフラ相当Capstoneでは、個別のコマンド暗記ではなく、攻撃、検知、修正、SLO、リリース、rollback、postmortemを一つの成果物にできるための判断軸を学びます。

具体例

  • 具体例: capstone profileで観測、edge proxy、Redisを同時に起動する
  • 具体例: S15を実施し、脆弱性レポート、postmortem、release checklistを作る
  • 具体例: Cloud/IAM、SBOM、SAST/DAST/SCA、secret rotationをS16-S33の高度課題へ接続する

到達目標

  • 攻撃、検知、修正、SLO、リリース、rollback、postmortemを一つの成果物にできる
  • CrowdStrike/Cloudflare/Palo Alto級のセキュアインフラ面接で説明できる証跡を作る
  • 高度領域をS16-S33のroadmap、Issue、PR単位へ分解できる

カバー領域

  • cloud security
  • supply chain
  • release engineering
  • incident leadership
  • executive reporting

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p9 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。
  • 実cloud、BGP、CDN、組織policyの変更は行わず、設計レビューとサンプル証跡で練習する。

Hands-on Flow

  • capstone profileで観測、edge proxy、Redisを同時に起動する
  • S15を実施し、脆弱性レポート、postmortem、release checklistを作る
  • Cloud/IAM、SBOM、SAST/DAST/SCA、secret rotationをS16-S33の高度課題へ接続する

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P10に進む前に、複数領域の証跡を一つのportfolioとして説明できる」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • 大手セキュアインフラ相当Capstoneの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • executive summary、technical report、PR、CI結果、postmortemが揃っている
  • セキュリティ、SRE、バックエンドの判断を一つのtimelineで説明している
  • 次に実装すべきcloud/IaC/supply chain課題がIssue化されている
