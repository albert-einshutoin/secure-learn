Kubernetes Production・Platform Security

P13に進む前に、Kubernetesを単なるデプロイ先ではなくplatform control planeとして説明できる

SREWhitehatBackend

抽象的に何を学ぶか

Kubernetes productionでは、manifestを書くことではなく、clusterが不正な変更を拒否し、障害時に安全に縮退し、upgradeしてもサービス契約を壊さない状態を作ります。

具体例

  • Admission: privileged podやlatest imageを拒否するpolicyを設計する
  • RBAC/NetworkPolicy: admin以外がsecretを読めず、pod間通信も必要最小限にする
  • Upgrade: CRD/Operator/API deprecationを事前に検出し、rollback条件を決める

到達目標

  • Helm/Kustomizeで環境差分を管理し、diff reviewできる
  • RBAC、PodSecurity、Admission、NetworkPolicyをplatform guardrailとして説明できる
  • HPA/VPA、service mesh、multi-cluster、upgradeの運用リスクを整理できる

カバー領域

  • Helm
  • Kustomize
  • Operator
  • CRD
  • Admission Controller
  • RBAC
  • NetworkPolicy
  • PodSecurity
  • HPA
  • VPA

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p12 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。

Hands-on Flow

  • 既存k8s/baseを読み、probe/resource/securityContext/NetworkPolicy/HPAを説明する
  • k8s_static_checkで最低限の安全条件を検査する
  • Admission/RBAC/Helm/Kustomize/Operatorを導入する場合のPR review checklistを作る

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P13に進む前に、Kubernetesを単なるデプロイ先ではなくplatform control planeとして説明できる」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Kubernetes Production・Platform Securityの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • RBAC/NetworkPolicy/PodSecurity/Admissionの役割分担を説明している
  • manifest diff reviewで見るべき危険差分を列挙している
  • cluster upgrade、CRD変更、rollbackのrunbookがある
