P6 / Independent 2 / phase6-release

Kubernetes・IaC・安全なリリース

P7に進む前に、releaseとsecurity boundaryを同じPRで説明できる

SREBackend

抽象的に何を学ぶか

Kubernetes・IaC・安全なリリースでは、個別のコマンド暗記ではなく、ComposeからKubernetes manifestへ運用境界を翻訳できるための判断軸を学びます。

具体例

  • 具体例: release profileを起動し、edge proxyからapp healthを確認する
  • 具体例: k8s_static_checkでKubernetes manifestの安全条件を確認する
  • 具体例: canary/rollbackの判断基準をrunbookへ落とす

到達目標

  • ComposeからKubernetes manifestへ運用境界を翻訳できる
  • readiness/liveness/startup probe、HPA、NetworkPolicyを説明できる
  • edge proxy経由でrelease/rollback観点を練習できる

カバー領域

  • Kubernetes
  • NetworkPolicy
  • rollback
  • canary
  • Terraform design

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p6 で起動対象profileを確認する。
  • Docker DesktopまたはDocker Engineが起動し、docker compose config -q が通る。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。
  • 実cloud、BGP、CDN、組織policyの変更は行わず、設計レビューとサンプル証跡で練習する。

Hands-on Flow

  • release profileを起動し、edge proxyからapp healthを確認する
  • k8s_static_checkでKubernetes manifestの安全条件を確認する
  • canary/rollbackの判断基準をrunbookへ落とす

Docker実行

scripts/learning_phase.sh config p6
scripts/learning_phase.sh start p6

curl -s http://localhost:8080/health

sed -n '1,220p' docs/runbooks/kubernetes-operations.md

観測ポイント

  • 開始時刻、対象profile、実行コマンド、終了時刻を記録する。
  • service health、ログ、メトリクス、テスト結果のどれで進捗を判断するかを決める。
  • 次フェーズ判定「P7に進む前に、releaseとsecurity boundaryを同じPRで説明できる」を満たした根拠を残す。

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Kubernetes・IaC・安全なリリースの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • probe、resource、securityContext、NetworkPolicyを説明している
  • rollback条件とmigration互換性の確認項目がある
  • IaC plan reviewで見るべき差分を列挙している
