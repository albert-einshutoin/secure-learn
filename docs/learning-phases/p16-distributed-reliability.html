Distributed Systems・Data Reliability

P17に進む前に、部分失敗前提のbackend/SRE設計ができる

SREBackend

抽象的に何を学ぶか

分散システムでは、1回のHTTP成功ではなく、重複、遅延、順序入れ替わり、部分失敗が起きてもデータとSLOを守る設計を学びます。

具体例

  • idempotency: retryで同じ支払い/作成処理が2回実行されないようkeyを持つ
  • backpressure: queue backlogが増えた時に429、shed load、consumer scaleのどれを選ぶか判断する
  • replication/sharding: consistencyとavailabilityのどちらを犠牲にするかを明示する

到達目標

  • Kafka/PubSub/Temporal/queueのfailure modeを説明できる
  • retry/backoff/idempotency/backpressureをAPI contractに落とせる
  • leader election、consistency、sharding、replicationの運用リスクを説明できる

Hands-on Flow

  • Redis profileでqueue/backlogの代替演習を行う
  • 同じrequestを複数回送ってidempotency設計をレビューする
  • consumer停止、retry storm、backlog増加のrunbookを書く
