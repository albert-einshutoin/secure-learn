Distributed Systems・Data Reliability

P17に進む前に、部分失敗前提のbackend/SRE設計ができる

SREBackend

抽象的に何を学ぶか

分散システムでは、1回のHTTP成功ではなく、重複、遅延、順序入れ替わり、部分失敗が起きてもデータとSLOを守る設計を学びます。

具体例

  • idempotency: retryで同じ支払い/作成処理が2回実行されないようkeyを持つ
  • backpressure: queue backlogが増えた時に429、shed load、consumer scaleのどれを選ぶか判断する
  • replication/sharding: consistencyとavailabilityのどちらを犠牲にするかを明示する

読み方

初学者の見方

  • 最初に到達目標とカバー領域を読み、知らない用語を一つずつ潰す。
  • Docker実行のconfig、start、statusを順に実行し、どのprofileが動くか確認する。
  • 合格証跡の先頭項目「retry/backoff/idempotencyのAPI設計メモがある」を残してから次フェーズへ進む。

経験者の深掘り

  • 自分の既存業務と照らし、欠けている運用、セキュリティ、テストの観点を洗い出す。
  • 実行コマンドを単に通すだけでなく、失敗時のログ、metric、rollback条件を確認する。
  • 次フェーズ判定「P17に進む前に、部分失敗前提のbackend/SRE設計ができる」をレビュー可能な証跡として説明する。

レビューで見ること

  • 前フェーズの未完了を放置せず、証跡、判断理由、残リスクを揃える。
  • Whitehat/SRE/Backendのどの役割で何を学んだかを一文で説明する。
  • 本番へ持ち込む場合の追加権限、owner、承認、監視、rollbackを定義する。

事前準備

  • 前フェーズの合格証跡を確認し、未完了のまま次へ進まない。
  • scripts/learning_phase.sh config p16 で起動対象profileを確認する。

安全境界

  • 許可されたローカルDockerラボだけを対象にする。
  • 外部IP、第三者service、実本番accountにscan、payload、負荷を向けない。
  • 負荷試験は小さい値から始め、SLO悪化を確認したら停止して復旧を記録する。

Hands-on Flow

  • Redis profileでqueue/backlogの代替演習を行う
  • 同じrequestを複数回送ってidempotency設計をレビューする
  • consumer停止、retry storm、backlog増加のrunbookを書く

よくある失敗

  • profileを理解しないまま起動し、どのserviceを検証しているか説明できない。
  • コマンドが通った事実だけを残し、何を守ったか、何を観測したかを書かない。
  • 設計だけで完了扱いにし、実行可能な証跡または安全上の制約を明記しない。

セルフレビュー

  • Distributed Systems・Data Reliabilityの到達目標を、Whitehat/SRE/Backendのどの観点で達成したか説明できるか。
  • 失敗した場合に最初に見るログ、テスト、メトリクスを言えるか。
  • 本番相当へ進める時の追加権限、承認、rollback、ownerを定義できるか。

合格証跡

  • retry/backoff/idempotencyのAPI設計メモがある
  • queue backlogとSLO影響を説明している
  • consistency、replication、shardingの選択理由がある
